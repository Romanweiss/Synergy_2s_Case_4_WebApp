Подробное описание процесса решения задачи (разработка с нуля)
1. Анализ постановки задачи
- Требовалось выполнить две части:
  1) аналитическую — обзор рыночных WEB-систем и вариантов их использования в компании;
  2) практическую — реализовать WEB-приложение с базой данных.
- Дополнительно определено, что стек Delphi/IIS/MSSQL можно заменить на Python + PostgreSQL.

2. Проектирование архитектуры решения
- Выбрана клиент-серверная WEB-архитектура:
  - клиент (Swagger/curl/браузер);
  - backend (FastAPI);
  - база данных (PostgreSQL).
- Определен базовый домен «Туризм» с сущностями: туры, услуги, заказы.

3. Проектирование базы данных
- Создан SQL-скрипт `sql/init.sql`.
- Спроектированы таблицы:
  - `tours` — каталог туров;
  - `services` — каталог дополнительных услуг;
  - `orders` — заявки клиентов.
- Для таблиц заданы ключи и ограничения:
  - первичные ключи во всех таблицах;
  - внешние ключи из `orders` в `tours` и `services`;
  - `CHECK`-ограничения для числовых полей.

4. Реализация backend (FastAPI)
- Создано API с маршрутами:
  - `GET /health`;
  - `GET /tours`;
  - `GET /services`;
  - `POST /orders`;
  - `GET /orders`.
- Добавлена валидация входных данных через Pydantic-схемы.
- В `POST /orders` реализована проверка существования и активности тура/услуги.

5. Контейнеризация и развёртывание
- Настроены `Dockerfile` и `docker-compose.yml`.
- В compose описаны два сервиса:
  - `db` (PostgreSQL);
  - `api` (FastAPI + Uvicorn).
- Инициализация схемы БД выполняется автоматически через `sql/init.sql`.

6. Аналитическая часть
- Подготовлен файл `CASE_5_ANALYTICAL_REVIEW.md`.
- В обзоре рассмотрены классы рыночных WEB-систем:
  - платформы бронирования;
  - CRM-системы;
  - ERP/back-office решения;
  - системы документооборота.
- Описаны возможности и практические сценарии использования в компании.

7. Документирование
- Обновлен `README.md`:
  - описание кейса;
  - соответствие условию;
  - структура БД и API;
  - инструкция запуска и проверки.
- Сформирован текущий файл `task_solution_description.txt` для письменного отчета.

Структура программы и ответственность компонентов
1. `app/main.py`
- Точка входа FastAPI-приложения.
- Подключает все API-роутеры.

2. `app/db.py`
- Инициализирует подключение к PostgreSQL через SQLAlchemy.
- Предоставляет dependency `get_db()` для работы с сессией БД.

3. `app/models.py`
- Описывает ORM-модели `Tour`, `Service`, `Order`.
- Определяет связи и структуру данных приложения.

4. `app/schemas.py`
- Описывает схемы запросов и ответов API.
- Выполняет валидацию входных данных.

5. `app/routers/*.py`
- `health.py` — проверка доступности сервиса.
- `tours.py` — чтение каталога туров.
- `services.py` — чтение каталога услуг.
- `orders.py` — создание и получение заказов.

6. `sql/init.sql`
- SQL-описание структуры БД и стартовых данных.
- Используется при первичной инициализации PostgreSQL.

7. `CASE_5_ANALYTICAL_REVIEW.md`
- Аналитический обзор рыночных WEB-систем и их применимости в компании.

8. `README.md`
- Руководство по запуску и проверке проекта.

Итог
Разработано WEB-приложение в тематике «Туризм» на стеке Python + PostgreSQL, а также подготовлена аналитическая часть по рыночным WEB-системам и их использованию в компании. Решение полностью покрывает требования кейса в выбранном технологическом варианте.
