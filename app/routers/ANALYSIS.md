## Кейс-задача №4 — Анализ WEB-информационных систем на рынке (предметная область «Туризм»)

## 1. WEB-архитектура информационных систем
WEB-приложение — это информационная система, доступ к которой осуществляется через браузер или мобильное приложение по протоколам HTTP/HTTPS.  
Типовая WEB-архитектура строится по клиент–серверной модели:

- **Клиент**: браузер/мобильное приложение (интерфейс пользователя).
- **Сервер приложений**: бизнес-логика, API, авторизация, обработка запросов.
- **База данных**: хранение и обработка данных (туры, заявки, клиенты, оплаты и т.д.).

Ключевые преимущества WEB-архитектуры для компании:
- доступ из любой точки (удаленная работа, филиалы);
- централизованное хранение данных и единый источник правды;
- быстрые обновления без установки на ПК пользователей;
- интеграции через API (платежи, карты, CRM, аналитика);
- масштабирование (разделение БД и приложения, балансировка нагрузки).

---

## 2. Системы на рынке, построенные по WEB-архитектуре (туризм)

В туристической сфере WEB-архитектура является стандартом, так как взаимодействуют разные роли:
- клиент (выбор тура, заявка);
- менеджер (обработка заявок, подбор вариантов);
- руководитель (контроль продаж, аналитика);
- бухгалтерия (оплаты, документы);
- партнеры (отели, авиакомпании, принимающие стороны).

Ниже приведены основные классы WEB-систем и варианты применения.

### 2.1 Онлайн-агрегаторы и платформы бронирования
**Назначение:** поиск и сравнение предложений, бронирование, оплата, выдача подтверждений.  
**Типовые функции:**
- каталог туров/отелей/перелётов;
- фильтры и сортировки, рекомендации;
- бронирование и оплата онлайн;
- личный кабинет клиента;
- уведомления по e-mail/мессенджерам.

**Варианты использования в компании:**
- источник лидов и заявок;
- канал онлайн-продаж без участия менеджера на раннем этапе;
- витрина актуальных предложений.

**Архитектурные особенности:**
- высокие нагрузки, кэширование, CDN;
- интеграции с внешними поставщиками (GDS/отельные базы/платежи);
- защищенные соединения (HTTPS), авторизация и защита персональных данных.

---

### 2.2 Back-office системы туроператоров и турагентств (операционный контур)
**Назначение:** управление продуктом и операциями компании.  
**Типовые функции:**
- ведение туров, цен, квот, доступности;
- управление заявками/заказами (OMS);
- работа с клиентами и документами;
- статусы заявок, подтверждения, счета.

**Варианты использования в компании:**
- единое рабочее место менеджера;
- контроль статусов заказов и взаимодействие с клиентом;
- снижение ошибок за счет единой базы данных и бизнес-правил.

**Архитектурные особенности:**
- роли и права доступа;
- журналирование действий;
- интеграции с CRM/бухгалтерией/документооборотом.

---

### 2.3 CRM для туризма (или универсальные CRM)
**Назначение:** управление клиентами и продажами.  
**Типовые функции:**
- карточка клиента и история обращений;
- воронка продаж и этапы сделки;
- сегментация клиентов и рассылки;
- аналитика продаж и источников заявок.

**Варианты использования в компании:**
- повышение конверсии (прозрачная воронка);
- повторные продажи и работа с лояльностью;
- контроль качества обработки заявок.

**Архитектурные особенности:**
- интеграция с сайтом/формами заявок/телефонией;
- API для обмена данными с back-office системой.

---

### 2.4 Электронный документооборот (EDMS/ECM) для туристических компаний
**Назначение:** хранение и согласование документов.  
**Типовые функции:**
- договоры, счета, акты, ваучеры;
- маршруты согласования;
- контроль версий, права доступа, архив.

**Варианты использования в компании:**
- ускорение согласований;
- централизованный архив документов;
- снижение рисков потери данных.

## 3. Оценка возможностей WEB-систем для туристической компании
WEB-архитектура в туризме особенно полезна из-за высокой динамики данных (цены/наличие/сезонность) и большого числа взаимодействий.

Критерии оценки WEB-систем:
1. **Доступность и удобство**: работа из браузера, мобильная поддержка.
2. **Роли и безопасность**: разграничение прав, аудит, защита данных.
3. **Интеграции**: платежные системы, внешние каталоги, CRM, аналитика.
4. **Масштабируемость**: рост нагрузки, разделение компонентов.
5. **Надежность**: резервное копирование, мониторинг, отказоустойчивость.

## 4. Реализованное WEB-приложение в рамках кейса (учебный MVP)
В рамках кейс-задачи разработано WEB-приложение с клиент–серверной архитектурой на тему «Туризм».

**Технологии:**
- Python + FastAPI (REST API)
- PostgreSQL (реляционная БД)
- Docker Compose (развертывание сервисов)

**Функции API:**
- `GET /health` — проверка доступности сервиса
- `GET /tours` — получение списка туров
- `POST /orders` — создание заявки на тур
- `GET /orders` — просмотр созданных заявок

**Как это соотносится с промышленными системами:**
- таблица `tours` выполняет роль справочника/каталога туров;
- таблица `orders` отражает переменную информацию (заявки клиентов);
- REST API соответствует типовой схеме интеграции WEB-систем (сайт/мобильное приложение → API → БД);
- Swagger/OpenAPI документация является стандартным способом описания API для дальнейшей интеграции.

## 5. Вывод
WEB-архитектура является базовым стандартом для туристических информационных систем, так как обеспечивает:
- централизованные данные и контроль процессов;
- удобный доступ для сотрудников и клиентов;
- масштабируемость и интеграции с внешними сервисами.

Разработанное учебное WEB-приложение демонстрирует ключевые элементы WEB-систем в туризме: каталог туров, прием заявок, хранение данных в реляционной БД и документированное API.
